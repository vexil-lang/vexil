# Vexil 1.0 Code Review — Performance, Memory, Safety, Lifecycle

**Date:** 2026-04-09
**Reviewer:** Hermes (mimo-v2-pro)
**Scope:** Full workspace (7 crates, 27,626 LOC)
**Severity Scale:** Critical / Important / Minor / Nit

---

## Executive Summary

The codebase is clean and well-structured. No unsafe blocks. SmolStr used
consistently for string interning. The runtime (BitWriter/BitReader) is
efficient with zero-copy reads. The main concerns are:

1. **No pre-allocated capacity in BitWriter** (frequent Vec growth)
2. **Clone patterns in hot paths** (typeck, lower)
3. **HashMap vs BTreeMap choice** in type registry
4. **Missing Send/Sync bounds** on public types
5. **Const evaluation overflow** (i64 arithmetic unchecked)

All are minor or important — no critical issues found.

## 1. PERFORMANCE

### 1.1 BitWriter — No Pre-allocation [Important]

**File:** `crates/vexil-runtime/src/bit_writer.rs`

```rust
pub fn new() -> Self {
    Self {
        buf: Vec::new(),  // ← starts at 0 capacity
        ...
    }
}
```

**Problem:** Every `BitWriter::new()` starts with 0 capacity. For typical
messages (64-1024 bytes), this means 3-4 reallocations (0→4→8→16→32→...→1024).

**Impact:** Medium. Reallocations in the hot encode path. Each grow requires
allocation + memcpy.

**Fix:** Add `with_capacity(n)` constructor. The generated code knows the
wire size from `wire_size_bits()` — use it.

```rust
pub fn with_capacity(bytes: usize) -> Self {
    Self {
        buf: Vec::with_capacity(bytes),
        ...
    }
}
```

Also consider changing `new()` to start with `Vec::with_capacity(64)` as a
sensible default. Most messages are under 64 bytes.

### 1.2 write_bits — One Bit at a Time [Minor]

**File:** `crates/vexil-runtime/src/bit_writer.rs:44`

```rust
pub fn write_bits(&mut self, value: u64, count: u8) {
    let mut v = value;
    for _ in 0..count {
        let bit = (v & 1) as u8;
        self.current_byte |= bit << self.bit_offset;
        self.bit_offset += 1;
        if self.bit_offset == 8 {
            self.buf.push(self.current_byte);
            self.current_byte = 0;
            self.bit_offset = 0;
        }
        v >>= 1;
    }
}
```

**Problem:** Writes one bit at a time. For a 3-bit field, this is 3 loop
iterations with branch-per-bit.

**Impact:** Low for sub-byte fields (usually small). Significant for
multi-byte writes that don't align (e.g., u12 at 5-bit offset).

**Fix:** Consider a bulk bit-copy optimization for cases where the value
fits in the remaining bits of the current byte. For the common case
(count ≤ remaining bits), no loop needed:

```rust
// Fast path: fits in current byte
if count <= (8 - self.bit_offset) {
    self.current_byte |= ((value as u8) & ((1 << count) - 1)) << self.bit_offset;
    self.bit_offset += count;
    if self.bit_offset == 8 {
        self.buf.push(self.current_byte);
        self.current_byte = 0;
        self.bit_offset = 0;
    }
    return;
}
// Slow path: spans byte boundary
```

### 1.3 read_bits — Same Pattern [Minor]

**File:** `crates/vexil-runtime/src/bit_reader.rs`

Same one-bit-at-a-time loop. Same optimization opportunity.

### 1.4 Vec Grows in Encoder Hot Path [Minor]

**File:** `crates/vexil-store/src/encoder.rs`

The encoder creates a `BitWriter::new()` for each `encode()` call. In a
batch scenario (encoding 1000 messages), this is 1000 Vec allocations.

**Fix:** Add a `reset()` method to BitWriter that clears the buffer but
keeps the allocation. Then callers can reuse:

```rust
pub fn reset(&mut self) {
    self.buf.clear();
    self.current_byte = 0;
    self.bit_offset = 0;
    self.recursion_depth = 0;
}
```

### 1.5 validate.rs — 75 String Allocations [Minor]

**File:** `crates/vexil-lang/src/validate.rs`

75 `format!`/`to_string()` calls for error messages. These are cold paths
(error reporting), so impact is negligible. But for large schemas with many
errors, it adds up.

**Fix:** None needed for 1.0. But if error performance becomes a concern,
consider lazy formatting (format only when the error is displayed).

## 2. MEMORY

### 2.1 TypeRegistry — Vec<Option<TypeDef>> [Important]

**File:** `crates/vexil-lang/src/ir/types.rs:49-54`

```rust
pub struct TypeRegistry {
    types: Vec<Option<TypeDef>>,
    by_name: HashMap<SmolStr, TypeId>,
    ...
}
```

**Problem:** Uses `Vec<Option<TypeDef>>` — `None` entries are "stubs" for
forward references. After lowering, stubs are resolved. But the `Vec` still
has `None` holes that waste space.

**Impact:** Minor for typical schemas (<100 types). Could matter for large
projects with many imports.

**Fix:** After all stubs are resolved, compact the Vec (remove `None`
entries, remap TypeIds). Or use a separate `Vec<TypeDef>` with no `None`.

### 2.2 Decl Names — Multiple Clones [Minor]

**File:** `crates/vexil-lang/src/lower.rs:355-369`

```rust
let name = match &decl.node {
    Decl::Message(d) => d.name.node.clone(),
    Decl::Enum(d) => d.name.node.clone(),
    ...
};
ctx.local_names.insert(name.clone());
```

The name is cloned from the AST into `local_names`, then cloned again into
the IR. Since `SmolStr` is copy-on-write and typically small (<24 bytes),
this is fine. But the `name.clone()` on line 364 could be `&name`.

### 2.3 typeck.rs — Clone in Loop [Minor]

**File:** `crates/vexil-lang/src/typeck.rs:22`

```rust
let decl_ids: Vec<TypeId> = compiled.declarations.clone();
```

Clones the entire declarations Vec just to iterate. Could use
`compiled.declarations.iter().copied()` instead.

### 2.4 validate.rs — Clone for Error Suggestions [Minor]

**File:** `crates/vexil-lang/src/validate.rs:506,1523`

```rust
find_closest_match(name.as_str(), all_types.clone().into_iter())
```

Clones the types Vec just to pass an iterator. Could use
`all_types.iter().cloned()` or pass a slice.

### 2.5 Box<dyn Error> in CodegenError [Nit]

**File:** `crates/vexil-lang/src/codegen.rs:72`

```rust
BackendSpecific(Box<dyn std::error::Error + Send + Sync>),
```

This is the standard pattern for extensible error types. No issue, just
noted for completeness. The `Send + Sync` bounds are correct.

## 3. CONCURRENCY & THREAD SAFETY

### 3.1 CompiledSchema — Not Send/Sync [Important]

**File:** `crates/vexil-lang/src/ir/mod.rs`

`CompiledSchema` contains `TypeRegistry` which contains `Vec<Option<TypeDef>>`.
All inner types are `Send + Sync`, so `CompiledSchema` should auto-derive
`Send + Sync`. But there's no explicit guarantee.

**Problem:** If someone adds a non-Send field (e.g., `Rc`), the compiler
won't catch it at the API boundary.

**Fix:** Add explicit bounds on the public API:

```rust
// In lib.rs
pub fn compile(source: &str) -> CompileResult
where
    CompiledSchema: Send + Sync,
{
    ...
}
```

Or add `#[derive(Send, Sync)]` or `unsafe impl Send for CompiledSchema {}`
with a comment explaining why it's safe.

### 3.2 No Parallel Compilation [Minor]

The compiler is single-threaded. For large projects, `compile_project`
could parallelize file loading and compilation. Not a bug, just a future
optimization opportunity.

### 3.3 BitWriter/BitReader — Not Shared [Info]

`BitWriter` and `BitReader` are designed for single-threaded use. This is
correct — encoding/decoding should never be shared across threads. The
types are `!Sync` by default (Vec interior), which is the right behavior.

## 4. SAFETY

### 4.1 No Unsafe Blocks [Excellent]

Zero `unsafe` blocks in the entire codebase (excluding test code). This is
exceptional for a systems-level compiler.

### 4.2 Const Evaluation — i64 Overflow [Important]

**File:** `crates/vexil-lang/src/lower.rs:1268-1282`

```rust
BinOpKind::Add => Some(left_val + right_val),
BinOpKind::Sub => Some(left_val - right_val),
BinOpKind::Mul => Some(left_val * right_val),
```

**Problem:** Integer overflow in const evaluation. `i64::MAX + 1` wraps
in debug mode (panics) or silently wraps in release mode. The value
is then stored as `ConstValue` and used in codegen.

**Impact:** A schema like `const X : i64 = 9223372036854775807 + 1`
would panic in debug or produce wrong values in release.

**Fix:** Use checked arithmetic:

```rust
BinOpKind::Add => left_val.checked_add(right_val),
BinOpKind::Sub => left_val.checked_sub(right_val),
BinOpKind::Mul => left_val.checked_mul(right_val),
BinOpKind::Div => {
    if right_val == 0 { None }
    else { left_val.checked_div(right_val) }
}
```

### 4.3 Recursion Depth — Good [Info]

`BitReader` and `BitWriter` track `recursion_depth` with a limit of
`MAX_RECURSION_DEPTH`. This correctly prevents stack overflow from
malicious recursive schemas. Well implemented.

### 4.4 LEB128 — Length Bounded [Info]

`read_leb128` takes a `max_bits` parameter and rejects encoding that
exceeds it. This prevents denial-of-service from extremely long LEB128
sequences. Well implemented.

### 4.5 String/Bytes Length Limits [Info]

`MAX_STRING_LENGTH` and `MAX_BYTES_LENGTH` constants cap decoded sizes.
Good defense against OOM from malicious input.

## 5. LIFECYCLE & RESOURCE MANAGEMENT

### 5.1 BitWriter — Consumed on Finish [Excellent]

```rust
pub fn finish(mut self) -> Vec<u8> {
    self.flush_to_byte_boundary();
    self.buf
}
```

The `finish()` method takes `self` by value, consuming the writer. This
prevents use-after-finish and ensures the buffer is complete. Well designed.

### 5.2 BitReader — Borrowed Data [Excellent]

```rust
pub struct BitReader<'a> {
    data: &'a [u8],
    ...
}
```

BitReader borrows data, doesn't own it. This enables zero-copy reads
(`read_bytes_ref`, `read_string_ref`) and prevents dangling references.
Well designed.

### 5.3 No Drop Implementations [Info]

No custom `Drop` impls in the codebase. All resources are owned by standard
types (Vec, String, HashMap) which have correct Drop. This is correct.

### 5.4 TypeRegistry — No Cleanup [Minor]

When compilation fails (returns `None`), the `TypeRegistry` is dropped
along with all allocated types. No leaks. But for the successful path,
the registry lives as long as the `CompiledSchema`. If the caller holds
the schema for a long time, all types stay allocated. This is by design
(the schema is the contract), but worth documenting.

## 6. API DESIGN

### 6.1 Public API Surface [Good]

The public API is well-tiered:
- Tier 1 (stable): `compile`, `CompiledSchema`, `CodegenBackend`
- Tier 2 (semi-stable): AST, pipeline stages
- Tier 3 (internal): lexer, parser, canonical

This is good for a 1.0 release.

### 6.2 Error Types [Good]

`thiserror` used consistently. `Diagnostic` carries span info, severity,
and human-readable messages. The new `edit_distance`/`did_you_mean` helpers
improve user experience.

### 6.3 CompileResult — Partial Success [Minor]

```rust
pub struct CompileResult {
    pub compiled: Option<CompiledSchema>,
    pub diagnostics: Vec<Diagnostic>,
}
```

**Problem:** A compilation can produce both `compiled: Some(...)` and
error diagnostics. The caller must check both. This is intentional
(warnings are diagnostics), but could be confusing.

**Fix:** Document clearly: "compiled is Some if and only if no Error
severity diagnostics were produced."

### 6.4 CodegenBackend — No Async [Info]

The `CodegenBackend` trait is synchronous. For CLI use, this is fine.
For LSP integration, async may be needed. Not a 1.0 concern.

## 8. SPECIFIC CODE PATTERNS

### 8.1 HashMap for Type Registry [Minor]

**File:** `crates/vexil-lang/src/ir/types.rs:31`

```rust
by_name: HashMap<SmolStr, TypeId>,
```

**Problem:** HashMap iteration order is non-deterministic. If any code
iterates `by_name` for codegen, output order may vary between runs.

**Fix:** Either use `BTreeMap` for deterministic iteration, or ensure
all codegen iterates over `declarations` (which is ordered) instead.

**Verification needed:** Check if codegen iterates `by_name` anywhere.

### 8.2 SmolStr — Excellent Choice [Info]

`SmolStr` stores up to 22 bytes inline (no heap allocation). For type
names, field names, and namespace segments, this covers 99% of cases.
Good choice.

### 8.3 Canonical Form — Deterministic [Good]

The canonical form sorts fields, types, and namespaces before hashing.
This ensures the BLAKE3 hash is deterministic regardless of source order.
Well implemented.
