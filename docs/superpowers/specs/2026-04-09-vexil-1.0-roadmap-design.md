# Vexil 1.0 Roadmap — Feature Decisions, Deferrals, and Rejected Alternatives

> Master design document for the v1.0 release. Records what we're building,
> what we're deferring, and what we rejected — with rationale for each.
> This is the reference for future RFCs and contributors.

**Date:** 2026-04-09
**Status:** Approved
**Supersedes:** Nothing (new document)
**Related:** `2026-03-30-language-gaps-design.md`, `2026-03-26-release-model-design.md`

---

## Context

Vexil-lang (Validated Exchange Language) is the schema definition language
for the Orix ecosystem. The C# ancestor project ("Axion") had a much richer
feature set including ECS primitives (entity, component, world, system),
CQRS (command, event), traits, macros, and a full expression language.

For vexil 1.0, we adopt a **pragmatic middle ground**: the language is a
data schema foundation that higher-level systems (malt, protocol crates)
build on. It must not fight those abstractions, but it doesn't encode them
directly.

**Design principle:** Additive, composable, codegen-driven, annotation-ready.

### Phase 2: Core Primitives

#### 2.1 Fixed-Point Types

**Add:** `fixed32` (Q16.16), `fixed64` (Q32.32)

| Type | Wire | Range | Precision |
|------|------|-------|-----------|
| `fixed32` | 32 bits, two's complement | ±32768.0 | ~4.8 decimal digits |
| `fixed64` | 64 bits, two's complement | ±2,147,483,648.0 | ~9.6 decimal digits |

**Rationale:** The Orix ecosystem is built on deterministic fixed-point arithmetic (Five Law #2: "All numbers as Q32.32 DFixed64"). Without built-in types, every schema needs a custom newtype wrapper. This is non-negotiable for 1.0.

**Codegen:**
- Rust: `fixed64` → custom `DFixed64` type (from `atom` or `vexil-runtime`)
- TypeScript: `fixed64` → custom `DFixed64` type
- Go: `fixed64` → custom `DFixed64` type

**Annotations:**
- `@varint` — valid on fixed32/fixed64? DECISION NEEDED. Probably no (varint is for integers, fixed-point has different semantics).
- `@zigzag` — NOT valid (two's complement is the encoding, zigzag would change semantics).

#### 2.2 Const Declarations

**Add:** `const NAME : type = value`

```vexil
const MAX_HEALTH : fixed64 = 100.0
const TICK_RATE : u32 = 64
const BUFFER_SIZE : u32 = 4096
```

**Rules:**
- Constants are compile-time values only. No wire impact.
- Usable in: constraints, array sizes, config defaults.
- Type must be a primitive or fixed-point type (not message/union/etc).
- Value must be a literal expression (no references to other constants in 1.0).
- Scope: schema-level. Accessible within the declaring schema.
- Cross-schema: exportable via `import { MAX_HEALTH } from my.schema`.

**Wire impact:** None. Constants disappear after compilation.

#### 2.3 Type Aliases (Non-Generic)

**Add:** `type Name = existing_type`

```vexil
type DFixed64 = fixed64
type Entity = uuid
type Tick = u64
type Duration = i64
```

**Rules:**
- Aliases are transparent — `DFixed64` and `fixed64` are the same type on the wire.
- Aliases cannot reference other aliases (no `type A = B` where B is an alias). Resolve to terminal type.
- Aliases are not newtypes (newtypes create distinct types; aliases don't).
- Aliases are not exported as distinct types in codegen — they expand inline.
- Circular aliases rejected at parse time.

**Wire impact:** Zero. Pure ergonomics.

#### 2.4 Constraint Language (Simple)

**Add:** `where` clauses on fields

```vexil
message Player {
    health @0 : fixed64 where value >= 0.0 && value <= 100.0
    name   @1 : string where len(value) > 0 && len(value) <= 255
    level  @2 : u16 where value in 1..999
    email  @3 : string where value contains "@"
}
```

**Syntax:** `where <expression>`

**Expression operators:**
- Comparison: `==`, `!=`, `<`, `>`, `<=`, `>=`
- Logical: `&&`, `||`, `!`
- Range: `value in low..high` (inclusive), `value in low..<high` (exclusive)
- Field reference: `value` refers to the field's value
- Built-in functions: `len(value)` for string/bytes/array length

**Semantics:**
- Constraints are **declarative** — they describe valid states.
- Constraints generate validation code in codegen backends.
- Vexil itself does NOT evaluate constraints (no interpreter).
- A schema with constraints compiles to code that validates on encode/decode.
- Constraints are **not** part of the wire format — they're compile-time contracts.

**What this does NOT support (deferred):**
- Cross-field constraints (`where amount <= balance`)
- User-defined functions in constraints
- Constraints on nested fields (`where address.zip.len() == 5`)
- Regex patterns (`where value matches "^[a-z]+$"`)

### Phase 3: Generic Foundation

#### 3.1 Generic Type Aliases

**Add:** `type Name<T> = <expression using T>`

```vexil
type TypedId<T> = u64
type Vec2<T> = array<T, 2>
type Vec3<T> = array<T, 3>
type Vec4<T> = array<T, 4>
type Quat<T> = array<T, 4>
type Mat3<T> = array<T, 9>
type Mat4<T> = array<T, 16>
```

**Rules:**
- Generic aliases expand at use site. `Vec3<fixed64>` becomes `array<fixed64, 3>`.
- Type parameter must appear on the right-hand side.
- No generic constraints (no `where T: Numeric`). Validity checked at instantiation.
- Nesting: `Vec3<fixed64>` is valid. `Vec3<Vec3<fixed64>>` is valid (becomes `array<array<fixed64, 3>, 3>`).

#### 3.2 Geometric Types (Built-In Parameterized)

**Add:** `vec2<T>`, `vec3<T>`, `vec4<T>`, `quat<T>`, `mat3<T>`, `mat4<T>`

These are **built-in** types (like `optional<T>`), not just aliases. Codegen generates semantic types.

**Valid T:** `fixed32`, `fixed64`, `f32`, `f64`

**Wire encoding:**
- Components written in order: x, y, z, w (column-major for matrices).
- No padding between components.
- No count prefix (fixed size from type).
- Byte-aligned if T is byte-aligned.

**Codegen:**
- Rust: `vec3<fixed64>` → `DVector3`, `vec3<f32>` → `[f32; 3]` or glam type
- TypeScript: `vec3<fixed64>` → `DFixed64Vector3`, `vec3<f32>` → `Float32Array`
- Go: `vec3<fixed64>` → `DVector3` struct

**Rationale:** Geometric types are ubiquitous in simulation/game protocols. Axion had them as keywords (`VEC2`, `VEC3`, etc.). Parameterizing over element type supports both deterministic (fixed-point) and standard (float) use cases.

#### 3.3 Fixed-Size Arrays

**Add:** `array<T, N>`

```vexil
message Header {
    magic   @0 : array<u8, 4>       # [0x56, 0x45, 0x58, 0x49]
    version @1 : array<u8, 2>       # major, minor
    hash    @2 : array<u8, 32>      # schema hash
}

message Color {
    rgba @0 : array<u8, 4>
}

message Matrix3x3 {
    data @0 : array<f32, 9>         # row-major
}
```

**Wire encoding:** Just T × N bits. No LEB128 count prefix.

**Rules:**
- N must be a positive integer literal or constant reference.
- Element type T follows same rules as `array<T>` (variable-length).
- Cannot nest with variable-length: `array<array<u8>, 3>` is valid, `array<array<u8, 4>>` is not (outer is variable-length, no — wait, that's fine). Actually both are fine.

#### 3.4 Set Type

**Add:** `set<T>`

```vexil
message Player {
    tags @0 : set<string>           # {"admin", "vip", "beta-tester"}
    permissions @1 : set<Permission>  # enum-based set
}
```

**Wire encoding:** LEB128 count + sorted elements (ascending, same rules as map keys).

**Rules:**
- T must be a valid map key type (integer, string, bytes, uuid, enum, flags).
- Elements are sorted on encode for determinism.
- Duplicates rejected on encode (or silently deduplicated — DECISION NEEDED).

### Phase 4: Wire Refinements

#### 4.1 Inline Bitfields

**Add:** `bits { name1, name2, ... }`

```vexil
message Header {
    flags @0 : bits { read, write, execute, admin, reserved1, reserved2 }
}

message PacketFlags {
    control @0 : bits { syn, ack, fin, rst }
    options @1 : bits { compress, encrypt, checksum }
}
```

**Wire encoding:** Exactly N bits (one per named flag), LSB-first packing. Identical to a `flags` declaration but inline and unnamed.

**Rationale:** Named `flags` declarations are good for shared types, but inline bitfields reduce boilerplate for protocol-specific flag fields.

#### 4.2 Deterministic Map Key Ordering (Spec Fix)

**Current state:** Spec says map keys MUST be sorted but sort order for custom enum/flags types is ambiguous.

**Fix:** Define canonical sort order for ALL valid key types:
- Integer types: ascending numeric (signed types use signed comparison)
- `string`, `bytes`: ascending lexicographic (UTF-8 / raw byte)
- `uuid`: ascending lexicographic (16-byte big-endian)
- `enum`: ascending by ordinal value
- `flags`: ascending by bit value (1 << ordinal)
- Newtypes: follow the inner type's sort order

**Wire impact:** None (already required by spec, just making it precise).

#### 4.3 Zero-Copy Byte Slices

**Scope:** Implementation change, not spec change.

**Current:** BitReader copies data for strings and byte arrays.
**Target:** Return `&[u8]` (Rust) / `Uint8Array` views (TypeScript) backed by the input buffer.

**Requires:** Lifetime tracking in BitReader API. This is a breaking API change to `vexil-runtime`.

**Wire impact:** None. Pure performance optimization.

#### 4.4 Spec Freeze

After phases 1-4 are complete:
1. Update spec from `0.1.0-draft-r2` to `1.0.0`
2. Freeze wire format — no changes without major version bump
3. Freeze Tier 1 API — `compile()`, `compile_project()`, `CompiledSchema`, `CodegenBackend`
4. All future changes require RFC process

## Rejected Alternatives

| # | Proposal | Decision | Rationale |
|---|----------|----------|-----------|
| R1 | Separate geometric types (`vec3_fixed`, `vec3_f32`) | Rejected | Parameterized (`vec3<T>`) is cleaner, composable, extensible |
| R2 | Geometric types as stdlib only | Rejected | Too common to require imports; built-in with stdlib re-exports |
| R3 | f32-only geometric encoding | Rejected | Orix requires deterministic fixed-point; both encodings needed |
| R4 | Entity/Component/World/System declarations | Rejected (for vexil) | Promoted to malt; vexil provides foundation, not architecture |
| R5 | Command/Event declarations | Rejected (for vexil) | CQRS is malt's concern; messages + constraints cover the use case |
| R6 | Full expression language (1.0) | Rejected | Simple constraints (comparison + logical) sufficient for 1.0; defer full |
| R7 | Generic constraints (`where T: Numeric`) | Rejected (for 1.0) | Adds complexity; check validity at instantiation instead |
| R8 | Macros (1.0) | Rejected | Huge surface area; defer to 1.1 with proper RFC |
| R9 | Mandatory BLAKE3 hash in wire envelope | Rejected | Optional is sufficient; some use cases don't need it |
| R10 | Compression integration (1.0) | Rejected | Transport layer concern; applications layer zstd/etc on top |
| R11 | Self-describing wire format | Rejected | Core design principle: schema = contract, no runtime type info on wire |
| R12 | Implicit type coercion | Rejected | Explicit conversions only; prevents silent bugs |

## Resolved Design Decisions

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | `@varint` on fixed-point? | **ALLOW** | Useful wire savings for small fixed-point values. LEB128 is canonical, determinism preserved. @varint changes encoding, not semantics. |
| 2 | Set dedup strategy? | **SILENT DEDUP** | Matches map key sorting behavior. Set semantics imply uniqueness. Rejecting adds encode-time cost. |
| 3 | Geometric codegen style? | **GENERIC WRAPPER** | `Vec3<T>` with named x/y/z fields. Type-safe, readable, From/Into for glam interop (feature-gated). |
| 4 | Const cross-references? | **SIMPLE ARITHMETIC** | Allow +, -, *, / between const refs. Topological sort for deps. Division by zero caught at compile time. Real need (e.g., TICK_DURATION = 1.0 / TICKS_PER_SEC). |
| 5 | Enum sub-byte backing? | **KEEP CURRENT** | u8/u16/u32/u64 only. Auto-width already handles sub-byte. Explicit backing is for externally fixed widths (always standard bytes). |
| 6 | Constraint opt-out? | **NO OPT-OUT** | Always validate. Constraints are the whole point. Start strict, relax later if perf data demands it. Adding @no_validate at 1.1 is non-breaking. |

## Implementation Order

```
Phase 1 (Gaps):         ~1 week   Small fixes, no spec changes
Phase 2 (Primitives):   ~3 weeks  Core additions, spec updates
Phase 3 (Generics):     ~2 weeks  Foundation, spec updates
Phase 4 (Wire):         ~2 weeks  Refinements, spec freeze
Phase 5 (Tooling):      ~2 weeks  LSP, error messages, docs
Phase 6 (Release):      ~1 week   Testing, compliance, publish
                                  ─────────────────────────
                                  Total: ~11 weeks
```

Each phase produces a publishable version. Phases can overlap where dependencies allow.
